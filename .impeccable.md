# Impeccable Design Context

# Content Hub — Population Matters

## Design Context

### Users

**Dan Davis** (Digital & Marketing Manager) and **Francesca Harrison** (Content Specialist). A two-person team running all content for a UK human rights and environmental charity. Daily use — this is the first tool they open in the morning.

Context of use: laptop, varied environments (home office, trains, co-working). Not casual browsing — active work sessions where they need to see what's happening, plan what's next, and move content through a workflow. They know the tool well. It should reward that familiarity with speed and clarity, not explain itself repeatedly.

Job to be done: at a glance, know what's going out and when. Act on what needs attention — draft a post, review a piece, shift a date, flag a concern. Get in and out without friction.

Emotional goal: calm confidence. Starting the day with a clear view of the pipeline should feel reassuring, not overwhelming. The interface should reduce cognitive load, not add to it.

### Brand Personality

**Purposeful. Warm. Direct.**

Population Matters is a rights-based charity — empowering, evidence-led, and warm. Not clinical, not corporate, not a generic NGO. The internal tool should feel like it belongs to the same organisation as the public-facing work: professional but human, with clear opinions about what matters.

Voice and tone in microcopy: friendly and specific. "Add content" not "Create new item". "In review" not "PENDING_REVIEW". "Nothing here yet — add your first piece" not "No records found". Small team, so the tool can have personality.

### Visual Direction

**Theme:** Light mode. Clean, warm-white backgrounds. Not clinical.

**Colour tokens (from `src/styles/app.css`):**

- `ocean-600` (`#0e4d63`) — primary action colour; buttons, active states, selected indicators
- `ocean-400` (`#1fb1c7`) — hover, interactive highlights
- `aqua-500` (`#00ffff`) — PM Bright Blue accent; use sparingly for emphasis only
- `graystone-*` — neutrals for text, borders, backgrounds; graystone-900 for body, graystone-200 for borders
- Status palette: `emerald` = published/success, `amber` = scheduled/warning, `sky` = in review/info, `rose` = blocked/danger

**Typography:** Neue Haas Grotesk Display / Neue Haas Grotesk as `font-sans`. Clear H1 > H2 > H3 hierarchy. Comfortable line-height. Semibold for labels and buttons, regular weight for body and metadata.

**Shape language:** Rounded corners (rounded-xl on buttons, rounded-3xl on menus/modals). Soft shadows (shadow-sm for cards, shadow-2xl for elevated panels). No sharp edges.

**Animation:** Subtle and purposeful. Slide transitions for panels (0.3s ease). Hover transitions on interactive elements. No decorative animation.

**Anti-references — never build this:**

- Clinical/sterile: pure white + grey-on-grey, hospital aesthetic
- AI slop defaults: cyan-on-dark, purple-to-blue gradients, glassmorphism
- Generic NGO: light blue + white that could be any charity
- Cards-on-cards visual noise: deeply nested surfaces, too many borders
- SCREAMING_STATUS_LABELS in uppercase
- Empty states that say "No data available"

### Design Principles

**1. The calendar is the hero surface.**
Everything in this tool orbits the content calendar. It is not a sidebar feature or a tab among equals — it is the primary view. Give it the most space, the best treatment, and the clearest visual language. All other views should orient users back to it.

**2. Glanceable above all else.**
Content items must communicate their type, status, and assignee without clicking in. If a user has to open an item to answer "what is this and where is it at?", the design has failed. Colour, iconography, and layout must carry that information immediately.

**3. Status colours do work — don't waste them on decoration.**
Ocean and aqua are not decoration. They signal action, state, and structure. Use them where they carry meaning (primary CTA, active navigation, status indicators). Do not use them for decorative section dividers or ambient colour blocks that don't tell the user anything.

**4. Warm, not corporate.**
Two people use this every day. It can have warmth, specificity, and personality. Empty states should be encouraging ("Quiet week — good time to plan ahead"). Labels should be readable prose. Error messages should be helpful, not terse. When in doubt, be warmer.

**5. Reduce cognitive load on context switches.**
Dan and Fran work across blog posts, social media, email campaigns, and donor comms simultaneously. The visual language must make it easy to orient when switching between types — distinct enough to register at a glance, consistent enough not to require re-learning. Tags, badges, and iconography should be stable across the whole app.

### Accessibility

WCAG 2.1 AA minimum. All interactive elements must have visible focus states (currently: `focus-visible:ring-2 focus-visible:ring-aqua-400`). Colour cannot be the only signal for status — always pair with a label or icon. Text contrast: graystone-900 on white passes easily; graystone-600 on white requires care (check at 14px). Reduced motion: respect `prefers-reduced-motion` for slide transitions.

### Stack Notes for Design Decisions

- **Tailwind 4.2** with `@theme` tokens — use token names (`ocean-600`, `graystone-200`) never raw hex in components
- **React 19** — transitions and deferred state are available for perceived performance
- **No component library** — all UI is bespoke; maintain the existing rounded/shadow language
- **Supabase Realtime** — collaborative indicators (who's editing, notification bell) are live; design for presence gracefully
