## Simplification Opportunities

### 1. app.jsx is the primary debt item

At 1976 lines with a single `ContentDashboard` function, app.jsx is doing far too much. Evident extraction candidates visible from the grep outline:

- **Navigation/routing** — `currentView`, `planTab`, `hasFeature` guard logic, hash-based routing via `syncFromHash` useEffect. Could be a `useNavigation` hook.
- **ManagerHub** — `_managers` state + `refreshManagers` + `handleManagerChange` are annotated with `// consumed when ManagerHub is extracted`. This is ready to extract now.
- **Bootstrap loading** — `loadBootstrapData` + `applyBootstrapData` + 3 useEffects that call them. Could be a `useBootstrap` hook.
- **Entry creation side-effects** — `createEntryFromIdea`, `buildEntryPrefillFromRequest`, `entryCreatedSideEffectsRef` pattern could be a `useEntryCreation` hook.
- **Modal state** — at least 6 modal open/close booleans managed at top level. A `useModalState` helper or moving modal state to feature hooks would reduce the surface.
- **Direct SUPABASE_API calls** — `app.jsx` calls `SUPABASE_API.fetchUserProfiles()` and `SUPABASE_API.logActivity()` directly, bypassing the hook layer. These should move into `useAdmin` or a dedicated `useUserProfiles` hook.

### 2. lib/supabase.ts needs domain splitting

At 2884 lines, `lib/supabase.ts` is the largest file in the project. It conflates:

- Supabase client init and auth
- 13+ entity-specific CRUD groups (entries, ideas, opportunities, requests, peaks, series, rapid responses, reporting periods, monthly reports, LinkedIn, influencers, guidelines, users)
- ~25 mapToDb / mapToApp transform functions
- Timestamp and enum mapping utilities

Suggested split: one file per domain entity (e.g., `lib/api/entries.ts`, `lib/api/reporting.ts`) plus a `lib/api/client.ts` for init. Transform functions could live alongside their entity file or in `lib/transforms/`.

### 3. EntryModal.jsx — 1866 lines, untyped, decomposable

The three internal render functions (`renderAssetNotes`, `renderApproverContent`, `renderCommentsSection`) should be extracted as proper components. They are large enough to warrant it:

- `renderCommentsSection` includes the full mention/comment system
- `renderApproverContent` includes the approval timeline, preview carousel, and social preview
- `renderAssetNotes` handles asset type conditional rendering

Additionally, draft autosave logic (the `setInterval` useEffect, `clearSavedDraft`, `restoreSavedDraft`, `showDraftRecovery` state) is a discrete concern that belongs in a `useDraftAutosave` hook.

Converting to TypeScript would also surface hidden type issues (the JSX file has no type checking at present).

### 4. EntryForm.tsx — 28+ top-level useState calls

The form uses 28+ individual `useState` calls rather than consolidated form state. This means every field change causes recalculation of all downstream `useMemo` values. Grouping related fields (e.g., partner fields, accessibility fields, assessment fields) into sub-objects would reduce re-render surface and make the state shape self-documenting.

### 5. Convention violations — direct Supabase calls in features

Three files call Supabase directly instead of routing through hooks or SUPABASE_API:

- `features/publishing/PlatformConnectionsView.tsx` — calls `getSupabase()` directly at 3 points
- `features/reporting/ReportingView.tsx` — calls `SUPABASE_API` directly
- `features/reporting/ReportInsightsView.tsx` — calls `SUPABASE_API` directly

The reporting views likely predate the hook pattern stabilising. A `useReporting` hook exists but the views bypass it for some operations.

### 6. Migration naming inconsistency

Migrations 001–014 use sequential numeric prefixes. Later migrations (`20260302212839_remote.sql` etc.) use timestamp prefixes. This makes ordering ambiguous — need to establish one convention and stick to it.

### 7. analyticsUtils.ts (742L) — test coverage unknown

Large pure-utility file. Needs explicit test coverage check. If analyticsUtils functions are untested, they're a correctness risk as the reporting system grows.

## Improvements I'd Recommend

These are not on any current list.

**1. EntryPreviewModal.jsx — verify it is not a duplicate of EntryModal**
Both exist in `src/features/entry/`. EntryPreviewModal may be a read-only subset of EntryModal, in which case it should be a prop-controlled render mode of EntryModal rather than a separate file. Deduplication would reduce surface significantly.

**2. CalendarView.tsx (928L) is too large for a container component**
It manages filter state, saved filters, view mode (month/week), bulk date shift, and upcoming deadlines. The filter panel + saved filters logic should be in a `useCalendarFilters` hook; bulk operations should be a `useBulkDateShift` hook. MonthGrid and WeekGrid are already extracted — the container is still the problem.

**3. EngagementView.tsx (892L) should follow the reporting wizard pattern**
LinkedIn analytics submission is large because it handles multi-step form state inline. The reporting wizard pattern (ReportingView → wizard steps) is already established. EngagementView predates it and should be refactored to use the same approach, or at minimum extracted into step sub-components.

**4. The `_managers` underscore-prefix is a code smell**
`const [_managers, setManagers]` in app.jsx uses underscore to signal "not yet consumed at this level." This is a workaround for deferred extraction. Once ManagerHub is extracted, this entire state block disappears from app.jsx.

**5. No integration/E2E tests**
81 tests are unit tests (Vitest). There are no Playwright or Cypress tests covering the entry creation → approval → reporting flow. For a workflow tool used by 18 staff, critical path E2E tests would catch regressions faster than manual testing.

**6. Realtime subscriptions not visible in audit**
CLAUDE.md mentions "Realtime subscriptions for collaborative features." These are not visible in the hook outlines. Worth confirming which entities have active subscriptions and whether cleanup (unsubscribe on unmount) is consistently handled — a common source of memory leaks in Supabase Realtime usage.

**7. feature-specs.md should be updated or archived**
All three features in the spec file appear to have been built. The file still reads as a forward-looking plan. Update it to reflect current state or move to a completed archive to avoid confusion.

**8. Fran adoption of Content Hub calendar is blocked**
Per org dynamics notes: the calendar was built but not adopted. A walkthrough guide (`docs/fran-onboarding.md`) targeting her specific workflow (drafting → pillar tagging → calendar placement) would have higher ROI than further feature additions until adoption is confirmed.
